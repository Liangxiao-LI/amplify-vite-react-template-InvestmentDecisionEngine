Architecture & Data Model

Enterprise GenAI Use-Case Decision Platform

A decision-support system. A user submits a structured GenAI use case; the platform combines three engines — deterministic rules, a supervised scoring model, and an LLM judge — to produce an explainable investment decision card, and an accountable human reviewer makes the final call. Below is the real data model, user lifecycle, and underlying AWS architecture from the current codebase.

Engine A
Deterministic Rules
7 policy rules via keyword match. Emits compliance flags and caps the recommendation (can lower, never raise).
Engine B
Supervised Scorecard
An interpretable additive model fit on-read from senior GoldenLabels. Outputs 5 dimension scores + contributions.
Engine C
LLM as Judge
Amazon Bedrock (Claude Haiku). Produces the summary, recommendation, controls, and policy citations.
Final
Human Reviewer
AI output is advisory. Approve / reject / request changes is a human act, stored separately from AI output.

Append-only
GoldenLabel grants create / read only — no update or delete. The training ground truth can never be silently rewritten.
Separation
Evaluation (AI output) and ReviewerDecision (human decision) are stored in separate tables for accountability and auditability.
02

Inside the Three Decision Engines

One submitted use case is assessed by three independent engines. Each has a different input, a different underlying logic, and a different basis of evidence for what it produces. Their outputs are then merged deterministically into a single Investment Decision Card — nothing is a black box, and every number traces back to a source. Read each column top-to-bottom: input → logic → output, with the dashed note showing what that engine's judgement is grounded in.

Engine A
Guardrail. Runs first and is fully deterministic. Its policy rules can only lower a recommendation (a floor), never raise it — compliance is non-negotiable and reproducible.
Engine B
Judgement. The five scores come straight from senior Golden Labels; each score ships with the exact feature contributions that produced it, so a reviewer can see why.
Engine C
Narrative. The LLM explains and recommends, but is fenced in by the rubric, the approved-policy allow-list, and Engine A's floor — advisory, never binding.

User Journey · Working with the Three Engines

Every role's path runs through the same three-engine core. The Admin governs the engines, the Requester feeds them, and Reviewers — standard and senior/expert — act on their output. The senior reviewer also closes the loop, turning decisions back into the training signal that improves Engine B. Follow the numbered steps: each starts from a role and touches the engine core.

Admin
Governs the engines up front: picks the active Bedrock model, the rubric, and the feature flags — including a kill-switch that disables evaluation entirely.
Reviewer split
Standard reviewers clear routine cases; when Engine A raises a SPECIALIST_REVIEW flag (high-impact domain / restricted data), the case routes to a senior / expert reviewer.
Flywheel
Senior reviewers record Golden Labels — the ground truth that trains Engine B. The more they review, the closer the scorecard tracks expert judgement.
04

AWS Architecture (Amplify Abstraction Removed)

Amplify Gen 2 translated into the underlying AWS services it actually provisions: Hosting → CloudFront + S3, Auth → Cognito, Data → AppSync + DynamoDB, Functions → Lambda. Simple CRUD is served by AppSync directly against DynamoDB; the two protected mutations run through Lambda.
